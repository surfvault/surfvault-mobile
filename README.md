# surfvault-mobile

React Native / Expo mobile app for [SurfVault](https://app.surf-vault.com). Distributed via TestFlight and the App Store as "SurfVault - Never Miss a Wave" (bundle ID `com.surfvault.mobile`).

System context: [`ARCHITECTURE.md`](../ARCHITECTURE.md). API: [`surfvault-api/`](../surfvault-api). Web: [`surfvault-web/`](../surfvault-web).

---

## Stack

- **React Native 0.81.5** on **Expo SDK 54**, TypeScript **5.9**
- **Expo Router** (file-based routing) with bottom tabs + a top-level stack
- **Redux Toolkit + RTK Query** (ported from web with TypeScript types)
- **NativeWind v4** (Tailwind for React Native) + `StyleSheet.create` where needed
- **react-native-auth0** with **expo-secure-store** for tokens (iOS Keychain / Android Keystore) and silent 401 refresh
- **pusher-js/react-native** for real-time
- **expo-notifications** for push (Expo Push service, not Pusher Beams)
- **react-native-maps** with `react-native-map-clustering`
- **expo-image-picker**, **expo-image**, **expo-media-library**, **expo-file-system**

## Setup

```bash
cd surfvault-mobile
npm install
cp .env.local.example .env.local   # if not present, see below for required keys
```

Dev `.env.local`:

```
API_BASE_URL=https://dev-api.surf-vault.com
AUTH0_DOMAIN=dev-auth.surf-vault.com
AUTH0_CLIENT_ID=<dev Native client id>
AUTH0_AUDIENCE=https://dev-api.surfvault.com
PUSHER_APP_KEY=<dev key>
PUSHER_CLUSTER=us2
GOOGLE_CLIENT_ID=<if ever used — not wired today>
ENVIRONMENT=dev
```

For EAS builds, env is injected from `eas.json` per profile — `.env.local` is only used for `expo start` / `expo run`.

## Commands

```bash
# Local dev build on a connected device (Xcode installed)
npx expo run:ios --device

# JS-only hot reload against an existing dev-client build
npx expo start --dev-client

# Re-prebuild native projects (required when app.config.ts plugins change)
npx expo prebuild --clean

# EAS development build (internal distribution)
eas build --profile development --platform ios

# Production build (TestFlight / App Store ready)
eas build --profile production --platform ios

# Submit to App Store Connect / TestFlight
eas submit --platform ios
```

## Conventions

- **Files:** PascalCase components, kebab-case routes (matches Expo Router's file conventions), camelCase utilities.
- **Routing:** top-level stack routes (`session/`, `user/`, `access/`, `account/`, `edit-profile/`, `manage-favorites/`, `conversation/`, `notifications/`, `break/`, `follow-stats/`) push **over** the tab bar. Always navigate with `useTrackedPush()` and go back with `useSmartBack()`. Don't call `router.back()` directly on top-level routes — it loses tab context.
- **Tab bar visibility:** `useTabBar().setTabBarVisible(false)` when a bottom sheet or full-screen overlay is open. Restore on unmount.
- **Styling:** NativeWind `className` in tab screens; `StyleSheet.create` in top-level routes and anywhere you need precise layout math (NativeWind has context quirks in Modals and deeply nested navigation).
- **Auth:** token storage + refresh handled inside `AuthProvider` + `customBaseQuery.ts`. Handlers don't touch tokens.
- **Redux / RTK Query:** same `tagTypes` as web. Cache invalidation mirrors web exactly — if you add an endpoint, pick an existing tag or discuss before adding a new one.
- **Coordinates from the API come as strings.** Always `parseFloat()` before using.
- **Date formatting:** when passing a date to the API as a calendar day, use the local-date helper (format `YYYY-MM-DD` from `date.getFullYear/getMonth/getDate`), not `toISOString` — ISO flips to UTC and causes off-by-one bugs for users west of UTC.

## Key flows

### Authentication

`src/context/AuthProvider.tsx` manages login, logout, credentials refresh. Token lives in `expo-secure-store` (Keychain / Keystore). `customBaseQuery.ts` reads the token synchronously and, on 401, calls the registered `tokenRefresher` which goes through Auth0's `getCredentials()` for a silent refresh.

Login uses `prompt: 'login'` to force the Auth0 account picker (prevents silent reuse of a previous session). Logout:

1. Fire-and-forget `PATCH /user/clear-push-token` to deregister the Expo token.
2. `clearCredentials()` from `react-native-auth0` (skips the browser modal).
3. `clearAuthToken()` in the custom base query.
4. `rootApi.util.resetApiState()` to purge all RTK Query caches.

### Push notifications (Expo Push)

Flow lives in `app/_layout.tsx`:

1. `NotificationPrimingModal` asks the user after onboarding (respects 7-day dismissal cooldown).
2. `Notifications.getExpoPushTokenAsync({ projectId })` on permission grant.
3. `PATCH /user/push-token` with the Expo token.
4. Backend (`notification` service) pushes via `expo-server-sdk` when a `notification` or `message` event fires for the user.
5. Tap-to-deep-link via `addNotificationResponseReceivedListener`. `data.screen` values: `notifications`, `messages`, `conversation`, `session`, `user`, `access`.

Badge count = `unreadMessageCount + unreadNotificationCount`, set via `Notifications.setBadgeCountAsync(total)` and refreshed on `AppState === 'active'`.

### Upload manager

`src/context/UploadContext.tsx` holds the active upload. `app/(tabs)/upload.tsx` creates the session, requests presigned URLs, then the context PUTs each file to S3 and calls `finalize` **per file** — so if the app is killed mid-upload, already-uploaded photos are still durable. `completeSurfMediaUpload` is called when done.

`UploadProgressPill` floats above the tab bar showing progress + a cancel button (with confirmation alert). Auto-dismisses 3 seconds after completion.

### Save to Photos

`src/helpers/saveToPhotos.ts` — requests `MediaLibrary` permission, fetches a presigned URL from `GET /media/photo/{photoId}/download-url`, downloads via `expo-file-system`, then `MediaLibrary.saveToLibraryAsync`, then cleans up the cache file. `savePhotosToCameraRoll(ids[], onProgress)` does a batch with progress callbacks.

### Account deletion (Apple requirement)

`app/account/index.tsx` surfaces:

- Current plan name (parsed from `chargebee_subscription_type`)
- Storage usage bar
- **Close Account** button — destructive red, calls `PATCH /user/request-deletion` after an alert confirming 30-day grace + data loss. Auto-logs out.
- **Cancel Deletion** button when `deletion_requested_at` is set — `PATCH /user/cancel-deletion`.

Account-wide banner: `src/components/PendingDeletionBanner.tsx` shows the countdown while a deletion is pending.

See [`DATA_DELETION.md`](../DATA_DELETION.md) for the full cascade.

## API response gotchas (vs. web)

Worth repeating — response shapes differ from what a web dev might expect:

- `getSelf` → `results.user` (nested, not `results`)
- Nearby breaks → `results.nearbyBreaks` (not `results.surfBreaks`)
- Nearby photographers → `results.nearbyPhotographers`
- Surf breaks search → `results.breaks`
- Map search → `results.searchContent` (mixed users/breaks)
- Get user → `results.photographer`
- Coordinates are strings — `parseFloat` before use

## Known gaps

- **RevenueCat / in-app purchases**: key placeholder in `app.config.ts`, no SDK wired. Today the Account screen hints at web (`app.surf-vault.com/plans`) for billing.
- **OTA updates via EAS Update**: not configured. Hotfixes require a new native build.
- **Android store submission**: build profiles exist but the app has not been submitted to Google Play.
- **Universal Links** (`apple-app-site-association`): not configured. Deep links work via `surfvault://` scheme and push notifications.
